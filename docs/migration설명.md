## 🛠️ Tortoise ORM & Aerich: DB 관리 체계 가이드

우리 프로젝트에서 DB를 다루는 방식은 단순히 테이블을 만드는 게 아니라, **"파이썬 코드가 실제 DB를 지배하는 구조"**입니다. 각 요소의 역할을 순서대로 살펴볼게요.

### 1. 세 가지 '스키마'의 관계 (핵심 개념)

먼저 우리가 코드를 짤 때 마주하는 세 가지 형태를 구분해야 합니다.

* **`models/*.py` (ORM 모델)**: **"우리의 이상향"**입니다.
    * 파이썬 코드로 정의한 DB의 원본 설계도 ( = ERD)입니다 
    * 모든 변화는 여기서 시작됩니다.
* **DB 스키마 (실제 테이블)**: **"냉혹한 현실"**입니다. 
    * EC2의 PostgreSQL 안에 실제로 존재하는 테이블들입니다.
    * ORM 모델과 이 현실이 일치해야 앱이 터지지 않습니다.
* **`dtos/*.py` Pydantic 스키마 (API 데이터)**: **"포장지"**입니다.
    * DB와는 직접 상관없습니다.
    * FastAPI가 클라이언트(FE)와 데이터를 주고받을 때 "데이터 형식이 맞는지" 검사하는 용도입니다.
    * 보통 ORM 모델의 필드를 본떠서 만듭니다.



---

### 2. Aerich: DB의 '깃허브(Git)'

**Aerich**는 우리 프로젝트의 **DB 버전 관리자** ( = DB의 Git )입니다. 
코드를 수정하면 `git commit`을 하듯이, DB 설계를 변경하면 Aerich를 통해 **마이그레이션(Migration)**이라는 기록을 남겨야 합니다.

---

### 3. 마이그레이션 실행 과정 및 주의사항 (Step-by-Step)

우리가 `models/`의 파일을 수정했을 때 실제 DB에 반영되기까지의 핵심 공정입니다.

#### **① `uv run aerich migrate --name <버전 이름>` (설계도 추출)**
* **역할**: 현재 파이썬 코드(`models/`)와 실제 DB 상태를 비교하여 차이점을 찾아냅니다.
* **결과**: `db/migrations/` 폴더 안에 **`.py` 마이그레이션 파일**이 생성됩니다.
* **특징**: 아직 DB는 변하지 않았습니다. "이런 공사가 필요함"이라는 **지침서**만 만든 상태입니다.

#### **② `db/migrations/*.py` (마이그레이션 파일)**
* **무조건** 바뀐 파일 확인해야한다.
* 이 파일은 **"공사 지침서"**입니다. 내부를 보면 `ALTER TABLE ...` 같은 SQL 쿼리가 파이썬 코드로 변환되어 기록되어 있습니다.
* **중요**: 이 파일은 DB의 역사를 기록하는 일지입니다.
  * 이 파일들만 있으면 나중에 서버를 새로 구축해도 똑같은 구조의 DB를 1초 만에 복구할 수 있습니다.

#### **③ `uv run aerich upgrade` (실제 공사 시작 및 주의사항)**
* **역할**: 위에서 만든 지침서(`.py` 파일)를 읽어서 **실제 DB에 적용**합니다.
* **결과**: 이때 비로소 EC2 DB에 새로운 컬럼이 생기거나 테이블 구조가 바뀝니다.
* **⚠️ 컬럼 수정 시 필수 체크리스트 (공사 전 필독):**
    1. **컬럼 삭제 (철거, DELETE):** `upgrade`를 실행하는 순간, 해당 컬럼에 들어있던 기존 데이터는 **즉시 영구 삭제**되며 복구가 불가능합니다.
        * 실행 전, 우리 코드 전체에서 이 컬럼을 참조하는 곳이 없는지 반드시 '전체 검색'을 마쳐야 합니다.
    2. **컬럼 추가 (증축, CREATE):** 기존 데이터가 이미 있는 테이블에 새 컬럼을 넣을 때는 **빈칸(NULL) 처리**가 핵심입니다. 
        * 비워둬도 되면 `null=True`를 설정합니다.
        * 반드시 값이 있어야 한다면 `default` 값을 지정해 기존 데이터들의 빈칸을 채워줘야 합니다. 
        * 그렇지 않으면 공사 중 에러가 발생하며 중단됩니다.
    3. **이름 변경 (교체, RENAME):** 필드 이름을 바꾸면 Aerich는 이를 '기존 것 삭제 후 새것 생성'으로 오해하여 데이터를 날릴 수 있습니다.
        * 마이그레이션 파일에 `RENAME` 쿼리가 제대로 들어갔는지 꼭 확인해야 합니다.



---

### 💡 핵심 요약

> "DB 공사는 **계획(Migrate) -> 검토(File 확인) -> 실행(Upgrade)**의 3단계로 진행됩니다.
> 특히 컬럼을 **삭제**할 때는 데이터가 영원히 사라지니 신중해야 하고, **추가**할 때는 기존 데이터의 빈칸을 어떻게 채울지(`null` 혹은 `default`)를 반드시 미리 결정해야 합니다. 
> 안전한 공사를 위해 `upgrade` 전에는 항상 마이그레이션 파일을 한 번씩 같이 확인합시다!"

---

### 4. 팀원들이 꼭 기억해야 할 요약 (Cheatsheet)


| 용어/명령어 | 한 줄 요약 | 비유 |
| :--- | :--- | :--- |
| **`models/*.py`** | 파이썬으로 짠 DB 원본 설계도 | **건축 설계도** |
| **`aerich migrate`** | 코드 변경점을 찾아 마이그레이션 파일 생성 | **공사 계획서 작성** |
| **`migrations/*.py`** | DB 변경 이력이 담긴 버전 관리 파일 | **공사 기록 일지** |
| **`aerich upgrade`** | 마이그레이션 파일을 실제 DB에 반영 | **실제 리모델링 공사** |
| **`Pydantic`** | API 요청/응답 시 데이터 검증 도구 | **택배 내용물 검수** |

---

### 💡 꼭 명심해야 할 한마디 (팁)
`models/` 폴더의 파일을 수정했다면, 반드시 **`migrate`로 계획서를 쓰고 `upgrade`로 공사를 마무리**해야 합니다. 그래야 우리가 짠 코드와 실제 DB가 따로 놀지 않아요. 특히 한글 데이터 처리를 위해 인코딩 설정을 바꿀 때 이 과정이 꼬이면 데이터가 깨질 수 있으니 주의합시다!"

## 1. '마이그레이션이 꼬였다'는 것은 어떤 상황인가?

### **상황 A: 설계도 작성 단계에서 꼬임 (Before Upgrade)**
* **현상:** `aerich migrate`를 했는데 내가 의도하지 않은 필드가 추가되거나 삭제되는 쿼리가 생성된 경우입니다.
  * 혹은 팀원들이 올린 마이그레이션 파일과 내 로컬 파일의 버전 번호가 충돌해서 에러가 날 때를 말합니다.
* **해결:** 아직 DB 공사(`upgrade`) 전이라면, 그냥 새로 생성된 `db/migrations/models/*.py` 파일을 **삭제**하고 다시 `migrate`를 실행하면 됩니다.

### **상황 B: 실제 공사 후 사고 발생 (After Upgrade)**
* **현상:** `aerich upgrade`를 완료했는데, 앱이 터지거나 잘못된 컬럼이 삭제된 것을 뒤늦게 발견한 경우입니다.
* **해결:** 이때 사용하는 것이 바로 **`aerich downgrade`**입니다.

---

## 2. `aerich downgrade`: 타임머신 타기

`downgrade` 명령어는 현재 DB의 스키마 버전을 한 단계 전으로 되돌리는 역할을 합니다.

* **동작 원리:** 마이그레이션 파일 안에는 공사를 진행하는 `upgrade` 로직뿐만 아니라, 공사를 취소하는 `downgrade` 로직(SQL)이 함께 들어있습니다.
  * 이 명령을 치면 DB 구조를 직전 상태로 복구합니다.
* **명령어:** `aerich downgrade`



---

## 3. ⚠️ 가장 중요한 질문: "컬럼이 삭제됐는데 되돌리면 데이터도 살아나나요?"

> **"구조(컬럼)는 돌아오지만, 데이터는 돌아오지 않습니다."**

* **구조 복구:** `downgrade`를 하면 삭제됐던 컬럼이 다시 생겨납니다.
  * DB 테이블 모양은 예전으로 돌아갑니다.
* **데이터 증발:** 하지만 `upgrade` 시점에 이미 삭제(Drop)된 데이터는 DB 엔진 수준에서 지워진 것이기 때문에, `downgrade`로 컬럼을 다시 만든다고 해서 사라진 데이터가 마법처럼 다시 채워지지는 않습니다.
  * 그 칸은 그냥 **비어있는(NULL) 상태**로 돌아올 뿐입니다.

> **결론:** 컬럼 삭제 마이그레이션은 '낙장불입'입니다. 데이터를 살리려면 마이그레이션 전의 **DB 백업본**을 복구하는 수밖에 없습니다.

---

## 4. 마이그레이션 사고 방지 가이드

1.  **계획서 검토:** `migrate` 후 생성된 `.py` 파일을 열어 `upgrade` 메서드 안에 `DROP COLUMN`이 들어있는지 반드시 확인한다.
2.  **데이터 백업:** 중요한 컬럼을 삭제하거나 수정하기 전에는 반드시 DB 스냅샷(백업)을 찍어둔다.
3.  **코드 선수정:** DB에서 컬럼을 지우기 전에, 해당 컬럼을 사용하는 모든 코드를 먼저 수정하거나 주석 처리하여 `upgrade` 직후 앱이 터지는 것을 방지한다.
4.  **파일 관리:** 마이그레이션 파일은 깃(Git)으로 엄격히 관리하며, 팀원 간의 버전 번호가 꼬이지 않도록 주의한다.

### 💡 요약 정리

> "**`downgrade`**는 DB의 모양을 이전으로 되돌려주는 타임머신입니다.
> 하지만 주의하세요! 컬럼을 삭제했다가 `downgrade`로 살려내도, 그 안에 들어있던 **데이터는 이미 삭제된 상태**입니다.
> 즉, 공사 설계도는 바꿀 수 있어도 이미 부숴버린 내용물은 돌아오지 않으니, 삭제 작업은 항상 백업 후에 진행합시다!"

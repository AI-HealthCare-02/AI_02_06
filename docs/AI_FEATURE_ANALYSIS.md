# Downforce AI 기능 분석서

작성일: 2026-04-13

## 개요

이 문서는 Downforce 프로젝트의 AI 기능을 **비용**, **개인정보**, **Tool Calling**, **Agentic** 4가지 요소로 분석한 결과입니다.

---

## 전체 시스템 차별점 요약

| 기능 | 비용 | 개인정보 | Tool Calling | Agentic |
|------|------|----------|--------------|---------|
| **병용금기 체크** | 무료 (DB) | 안전 (로컬) | X | X |
| **OCR 약품명 교정** | 무료 (DB) | 안전 (로컬) | X | X |
| **복용 현황 분석** | 저비용 (LLM 1회) | 안전 (익명화) | X | X |
| **의약품 정보 RAG** | 중비용 (LLM 2-3회) | 안전 (개인정보 無) | O | O |

## 기능별 상세 분석

### 1. 병용금기 체크

**4요소 분석:**

| 요소 | 평가 |
|------|------|
| 비용 | 무료 (DB 조회만) |
| 개인정보 | 안전 (서버 내부 처리, 외부 전송 없음) |
| Tool Calling | X (고정 파이프라인) |
| Agentic | X (LLM 판단 없음) |

**기술 흐름:**
```
[사용자 약품 A + B 입력]
        |
        v
[PostgreSQL: drug_interaction_cache 조회]
        |
        v
[결과: 금기/주의/안전 + 사유]
        |
        v
[프론트엔드에 직접 반환]
```

**차별점:** AI 할루시네이션 없는 "결정적(Deterministic)" 안전 판단

---

### 2. OCR 약품명 자동 교정

**4요소 분석:**

| 요소 | 평가 |
|------|------|
| 비용 | OCR API 비용만 (LLM 불필요) |
| 개인정보 | 안전 (처방전에서 약품명만 추출) |
| Tool Calling | X (고정 파이프라인) |
| Agentic | X (규칙 기반 매칭) |

**기술 흐름:**
```
[처방전 이미지 업로드]
        |
        v
[Clova OCR -> 텍스트 추출]
        |
        v
[pg_trgm 퍼지 매칭 -> 유사 약품 검색]
        |
   +----+----+
   |         |
유사도>=80%  유사도<80%
   |         |
   v         v
자동 교정   사용자 확인 요청
```

**차별점:** LLM 없이 한글 약품명 오타 교정 (비용 절감)

---

### 3. 개인 복용 현황 분석 (채팅)

**4요소 분석:**

| 요소 | 평가 |
|------|------|
| 비용 | 저비용 (LLM 1회, ~400토큰) |
| 개인정보 | 안전 (익명화 후 전송) |
| Tool Calling | X (로컬 처리 후 LLM은 문장 생성만) |
| Agentic | X (LLM이 판단하지 않음) |

**기술 흐름:**
```
[사용자: "나 약 잘 먹고 있어?"]
        |
        v
[로컬 의도 분류 -> "복용현황_확인"]
        |
        v
[로컬 DB 조회]
 - 복용 기록 (intake_logs)
 - 순응도 계산 (completed / total)
 - 누락 패턴 분석
        |
        v
[익명화된 분석 결과 생성]
 "순응도: 85%, 주 누락 시간대: 저녁"
 (약품명, 개인정보 제외)
        |
        v
[GPT-4o-mini -> 자연어 응답 생성]
        |
        v
"잘 드시고 계세요! 다만 저녁 복용이 가끔 누락되네요.
 알림 시간을 조정해보시는 건 어떨까요?"
```

**차별점:** 민감정보 보호 + 자연스러운 AI 응답

---

### 4. 의약품 정보 RAG Agent

**4요소 분석:**

| 요소 | 평가 |
|------|------|
| 비용 | 중비용 (LLM 2-3회, ~1000토큰) |
| 개인정보 | 안전 (개인정보 사용 안 함) |
| Tool Calling | O (LLM이 도구 선택) |
| Agentic | O (자율적 판단 및 다중 도구 조합) |

**기술 흐름:**
```
[사용자: "타이레놀이랑 게보린 차이가 뭐야?"]
        |
        v
[GPT-4o-mini Agent 판단]
 "두 약품 정보를 각각 검색하고 비교해야겠다"
        |
        v
[Tool Calling 결정] <- LLM이 스스로 선택
 - search_drug_info("타이레놀")
 - search_drug_info("게보린")
 - compare_drugs(["타이레놀", "게보린"])
        |
        v
[도구 실행 - 개인정보 없음]
 - 식약처 DB 조회 (drug_dictionary)
 - pgvector 의학 지식 검색 (RAG)
        |
        v
[GPT-4o-mini -> 결과 종합 응답]
```

**차별점:** 진정한 Agentic Tool Calling + Privacy 보장

## 개인정보 흐름

**LLM에 전송되지 않는 것:**
- 사용자 이름, 생년월일
- 구체적인 복용 약품명
- 기저질환 상세 내역
- 복용 기록 원본

**LLM에 전송되는 것:**
- 익명화된 분석 결과 ("순응도 85%")
- 일반적 건강 주의사항 ("간 관련 주의 필요")
- 일반 의약품 정보 질의 ("타이레놀 부작용")

## 최종 발표용 차별점

### 1. Privacy-First 설계
- 민감 의료정보 외부 전송 제로
- 개인정보보호법/의료법 준수 구조

### 2. 비용 효율적 하이브리드
- 안전 판단 (병용금기): DB 규칙 기반 -> 무료
- 자연어 처리: LLM 최소 사용 -> 70% 비용 절감

### 3. 결정적(Deterministic) 의약품 안전 검증
- 식약처 DUR 데이터 기반
- AI 할루시네이션 위험 차단
- pg_trgm 한글 퍼지 매칭

### 4. Agentic RAG 의약품 정보 검색
- GPT-4o-mini Tool Calling
- 자율적 도구 선택 및 다중 검색
- pgvector 의학 지식베이스
- 개인정보 없이 Agentic 구현

## 추가 분석: 진짜 Tool Calling이 필요한 곳 (2026-04-13 업데이트)

### RAG Agent에 대한 재고

일반적인 약품 정보 질문("타이레놀 부작용", "게보린과 타이레놀 차이")은 GPT-4o-mini가 이미 잘 알고 있음. RAG를 사용하면 오히려:
- LLM 호출 2-3회로 비용 증가
- 복잡도 증가
- 품질은 비슷하거나 약간 나은 정도

**결론: 일반 지식 질문에 RAG는 비효율적**

---

### Tool Calling의 진짜 가치

**LLM이 절대 혼자 못하는 것 = Tool Calling의 진짜 가치**

| LLM이 혼자 가능 | LLM이 절대 불가능 (Tool 필요) |
|----------------|------------------------------|
| 타이레놀 부작용 설명 | "내가 오늘 약 먹었어?" |
| 약품 비교 | "내 복용 기록 보여줘" |
| 일반 건강 조언 | "이 약 내 약이랑 괜찮아?" |
| 식약처 일반 정보 | "알림 시간 바꿔줘" |
| 건강보험 일반 정보 | "오늘 약 먹었다고 기록해" |

**핵심: 사용자의 개인 데이터에 접근하거나 시스템을 조작하는 것은 Tool 없이 불가능**

---

### 재설계된 Agentic 시스템

**시나리오 1: Tool이 필요 없는 경우**
```
User: "타이레놀 부작용 알려줘"
Agent: (Tool 불필요 판단) → LLM 직접 응답
비용: LLM 1회
```

**시나리오 2: Tool이 필요한 경우**
```
User: "오늘 아침 약 먹었어?"

Agent 판단: "복용 기록을 확인해야 한다"
     |
     v
Tool Call: check_intake_log(profile_id, date=today, time="morning")
     |
     v
Tool 결과: {"status": "MISSED", "scheduled_time": "08:00"}
     |
     v
Agent 응답: "아직 안 드셨네요! 아침 8시 약이 대기 중이에요."

비용: LLM 2회 (판단 + 응답)
```

**시나리오 3: 개인 맥락 + Tool Calling**
```
User: "타이레놀 먹어도 돼?"

Agent 판단:
  "1. 사용자 현재 복용약 확인 필요"
  "2. 병용금기 체크 필요"
     |
     v
Tool Calls:
  1. get_user_medications(profile_id)
  2. check_drug_interaction("타이레놀", user_drugs)
     |
     v
Tool 결과:
  - 현재 복용: [해열진통제 계열] (익명화)
  - 상호작용: 주의 필요
     |
     v
Agent 응답: "현재 드시는 약과 함께 복용 시
            위장 부작용 위험이 있어요. 주의하세요."

비용: LLM 2-3회
```

## 최종 포트폴리오 차별점 (2026 트렌드 반영)

### 1. Agentic Tool Calling
- LLM이 "사용자 DB 조회가 필요한지" 자율 판단
- 6개 도구: 복용약 조회, 기록, 알림 설정 등
- 일반 질문은 LLM 직접 응답 (비용 절약)
- 개인 맥락 필요 시에만 Tool Calling

### 2. Privacy-Aware Tool Design
- Tool 결과 최소화 (약품명 대신 카테고리)
- 민감정보 마스킹
- 병용금기 최종 판단은 로컬 DB (LLM에 안 보냄)

### 3. 비용 최적화 하이브리드
- 일반 질문: LLM 직접 (1회)
- 개인 맥락: Tool Calling (2-3회)
- 안전 판단: DB 규칙 (0회)

### 4. 한국 특화 OCR 파이프라인
- Clova OCR + pg_trgm 퍼지 매칭
- LLM 없이 약품명 자동 교정

## Input Harness & Custom Skills (2026-04-13 추가)

### 개념 정의

**Input Harness (입력 전처리)**: 사용자 입력이 LLM에 도달하기 전에 가공/분류/라우팅하는 레이어

**Custom Skills (사전 정의 워크플로우)**: 특정 패턴의 요청에 대해 최적화된 고정 워크플로우 실행

---

### Input Harness 기능

| 기능 | 설명 | 효과 |
|------|------|------|
| **Emergency Detector** | 위험 키워드 감지 ("과다복용", "쓰러졌") | 즉시 대응 |
| **Intent Router** | 의도별 최적 경로 분기 | LLM 호출 최소화 |
| **Drug Name Extractor** | 입력에서 약품명 추출 | 정확도 향상 |
| **Slot Filler** | 필요 정보 자동 채우기 | UX 개선 |

---

### Custom Skills 목록

| 스킬명 | 트리거 예시 | 동작 |
|--------|------------|------|
| `MorningCheckSkill` | "오늘 약 뭐 먹어야 해?" | 오늘 예정 약품 + 복용 여부 조회 |
| `WeeklyReportSkill` | "이번 주 복용 현황" | 7일 통계 + 순응도 계산 |
| `QuickRecordSkill` | "아침 약 먹었어" | 즉시 복용 기록 |
| `InteractionCheckSkill` | "이거 먹어도 돼?" | 현재 복용약과 병용금기 체크 |
| `ReminderSetupSkill` | "알림 시간 바꿔줘" | 알림 설정 대화 시작 |

---

### Input Harness 흐름

```
User: "타이레놀 먹어도 돼?"
    |
    v
[1. Emergency Check]
    |  "과다복용", "쓰러졌", "숨이 안 쉬어져" 감지?
    |
    +-- Yes --> 즉시 응급 안내 반환 (LLM 스킵)
    |           "119에 연락하세요!"
    |
    v No
[2. Intent Classification]
    |  규칙 기반 패턴 매칭
    |
    |  "먹어도 돼" --> INTERACTION_CHECK
    |  "먹었어" --> RECORD_INTAKE
    |  "뭐 먹어야" --> TODAY_SCHEDULE
    |  "현황/리포트" --> WEEKLY_REPORT
    |
    v
[3. Entity Extraction]
    |  약품명 추출: "타이레놀"
    |  시간 추출: null
    |
    v
[4. Skill Matching]
    |  Intent: INTERACTION_CHECK
    |  Entities: {drug: "타이레놀"}
    |
    |  --> InteractionCheckSkill 매칭!
    |
    v
[5. Route Decision]
    |
    +-- Skill 매칭됨 --> Skill 실행 (최적화된 경로)
    +-- 매칭 안됨 --> Agent (Tool Calling)
```

### Skill 실행 예시: InteractionCheckSkill

```
Input:

  profile_id: "xxx"

Execution (고정된 워크플로우):

  Step 1: 사용자 복용약 조회 (DB)
          --> [아스피린, 혈압약]

  Step 2: 병용금기 체크 (DB)
          --> 타이레놀 + 아스피린 = 주의

  Step 3: 건강 프로필 확인 (DB)
          --> 간질환 플래그 있음

  Step 4: 결과 조합
          --> {
               interaction: "주의",

               recommendation: "의사 상담 권장"

```

## 최종 포트폴리오 차별점 (완성)

### 1. Input Harness (전처리 레이어)
- 응급 상황 즉시 감지 (LLM 없이)
- 의도 분류 + 엔티티 추출 (규칙 기반)
- 최적 경로 라우팅으로 비용 절감

### 2. Custom Skills (사전 정의 워크플로우)
- 자주 쓰는 5가지 패턴 최적화
- LLM 1회로 비용 절감 (Agent 대비 50-66%)
- 예측 가능한 품질과 응답 시간

### 3. Agentic Tool Calling (Agent Fallback)
- 예측 불가 질문 대응
- LLM이 자율적으로 도구 선택
- 6개 도구: 복용약 조회, 기록, 알림 설정 등

### 4. Privacy-First 설계
- 응급 감지: 로컬 (무료)
- Skill 실행: 익명화된 결과만 LLM에 전달
- 병용금기 판단: DB 규칙 기반 (LLM 없음)

### 5. 비용 최적화 계층 구조
- 응급: 즉시 반환 (무료)
- Skill 매칭: LLM 1회
- Agent Fallback: LLM 2-3회
- 예상 비용 절감: 기존 대비 60-70%

### 6. 한국 특화 OCR 파이프라인
- Clova OCR + pg_trgm 퍼지 매칭
- LLM 없이 약품명 자동 교정

## 발표용 최종 멘트

> **"Harness로 분류하고, Skill로 최적화하고, Agent로 유연하게"**

> **"LLM이 혼자 대답할 수 있는 건 LLM에게,**
> **자주 쓰는 패턴은 Skill로 최적화,**
> **사용자 데이터가 필요한 건 Tool Calling으로,**
> **안전이 중요한 건 검증된 DB로."**

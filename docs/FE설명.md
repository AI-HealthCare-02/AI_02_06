# 프런트엔드(FE)의 전체 메커니즘

---

## 1. FE 보안: "신분증 관리와 입구 컷"

FE 보안의 핵심은 **적절한 권한을 가진 사람에게만 기능을 허용하는 것**입니다.

* **인증(Authentication)과 신분증(JWT):**
    * 카카오 같은 외부 인증을 통해 받은 고유 ID를 우리 DB의 `Account`와 연결하는 과정입니다.
    * **JWT(JSON Web Token)**는 서버가 발행한 **디지털 신분증**입니다.
    * 로그인 성공 시 백엔드(BE)가 넘겨준 이 신분증을 **Secure Storage(암호화 저장소)**에 안전하게 보관합니다.
    * 이후 모든 요청마다 이 신분증을 **API 헤더**에 실어 보냄으로써 내가 누구인지 증명합니다.


* **유효성 검증(Validation):**
    * 서버로 데이터를 보내기 전, 입구에서 쓰레기 데이터를 걸러내는 과정입니다.
    * 이미지 칸에 텍스트를 올리거나, 숫자 칸에 문자를 넣는 실수를 FE에서 즉시 차단하여 서버 부하를 줄입니다.

---

**DownForce** 프로젝트의 세션 관리 파트에서 팀원들이 가장 궁금해할 **Access Token과 Refresh Token의 협동 작전**을 추가하여 정리해 드릴게요. 이 부분은 보안과 사용자 편의성(UX) 사이의 균형을 잡는 핵심 로직입니다.

---

웹 환경으로 개발될 **DownForce** 프로젝트의 특성에 맞춰, 브라우저 보안의 핵심인 **HttpOnly 쿠키** 개념을 반영하여 내용을 수정했습니다. 팀원들이 웹에서의 토큰 관리 체계를 정확히 이해하도록 돕는 가이드입니다.

---

## 2. 데이터 운영: "기억(State)과 정책(Session)"

### **상태(State) 관리**
* **개념:** 웹 앱이 실행되는 동안 브라우저가 유지해야 할 **기억**입니다.
    * (예: 현재 로그인한 유저 정보, 분석 중인 처방전 이미지 등)
* **저장 위치:** 
    * **메모리(Memory):** 새로고침을 하면 사라지지만 속도가 매우 빠릅니다.
      * (React의 State, Recoil, Zustand 등 라이브러리 사용)
    * **Storage:** 브라우저를 껐다 켜도 남아야 하는 정보는 `LocalStorage`나 `SessionStorage`에 저장합니다.
      * 단, 보안이 중요한 정보는 이곳에 평문으로 저장하지 않습니다.

---

### **세션(Session) 관리: "Access Token vs Refresh Token (Web Standard)"**
웹 브라우저에서는 보안을 위해 **유효기간이 짧은 활동용 신분증**과 **삼중 잠금 장치가 된 금고**를 함께 사용합니다.

| 구분 | Access Token (활동용 신분증) | Refresh Token (재발급용 마스터키) |
| :--- | :--- | :--- |
| **비유** | **놀이공원 자유이용권 팔찌** | **은행 금고에 보관된 인감도장** |
| **유효기간** | 매우 짧음 (예: 30분 ~ 1시간) | 매우 길음 (예: 2주 ~ 1달) |
| **저장 위치** | **브라우저 메모리 (변수)** | **HttpOnly + Secure + SameSite 쿠키** |
| **보안 특징** | XSS 공격 노출 위험으로 수명을 짧게 제한 | **탈취 및 위조가 불가능한 삼중 보안** |


#### **왜 웹에서는 이렇게 삼중 보안을 하나요?**
* **HttpOnly (XSS 방어):** 자바스크립트가 쿠키를 읽는 것을 원천 차단합니다. 
  * 해커가 악성 스크립트를 심어도 우리 신분증을 복사해서 가져갈 수 없습니다.
* **SameSite=Strict/Lax (CSRF 방어):** 사용자가 로그인된 상태로 해커가 만든 낚시 사이트에 접속했을 때, 브라우저가 자동으로 쿠키(신분증)를 보내는 것을 막습니다. 
  * **우리 서비스(Same Site) 안에서만 신분증을 사용하겠다**는 강력한 선언입니다.
* **Secure (도청 방어):** 오직 암호화된 통신(`HTTPS`) 환경에서만 신분증을 주고받아, 네트워크 중간에서 정보를 가로채는 것을 방지합니다.

#### **토큰 재발급 흐름 (Silent Refresh in Web)**
1. **요청:** FE가 메모리에 든 Access Token을 헤더에 실어 API 요청을 보냅니다.
2. **만료:** 서버가 "신분증 만료(401 Unauthorized)"를 응답합니다.
3. **갱신 요청:** FE가 재발급 API를 호출합니다.
   * 이때 **브라우저가 자동으로** 금고(쿠키)에 든 **Refresh Token**을 꺼내 서버로 보냅니다.
   * (이때 `SameSite` 설정 덕분에 오직 우리 서버로만 안전하게 전송됩니다.)
4. **새 신분증 발급:** 서버가 신분증을 확인하고 새 Access Token을 응답합니다.
5. **재시도:** 인터셉터가 새 토큰을 메모리에 업데이트하고, 실패했던 요청을 다시 보내 성공시킵니다.

## 3. 통신과 피드백: "전화기(API)와 알림(Toast)"

* **API 통신:**
    * FE와 BE 사이의 **전화기**입니다. 
      * 효율적인 통신을 위해 `Dio`나 `Axios` 같은 라이브러리를 사용합니다.
    * **인터셉터(Interceptor):** 모든 통신 시 자동으로 JWT 신분증을 동봉하는 '자동 발송' 기능입니다.
    * **타임아웃:** 서버 응답이 너무 늦어질 때 무한정 기다리지 않도록 끊어주는 안전장치입니다.

* **Toast:**
    * 화면 하단에 잠시 떴다 사라지는 가벼운 피드백입니다. 
    * 사용자 흐름을 방해하지 않고 정보를 전달할 때 사용합니다.


---

## 4. 아키텍처의 이해: "무대(Browser)와 배우(FE)"

* **브라우저 (실행 환경):**
    * FE 코드가 실행되는 **엔진**이자 **무대**입니다.
    * **렌더링 엔진:** HTML/CSS로 그림을 그립니다.
    * **JS 엔진:** 우리가 짠 로직을 해석하고 실행합니다.
    * **Web API:** 브라우저가 FE에 빌려주는 도구함(저장소, 위치 정보 등)입니다.
    

* **프런트엔드 (프로그램):**
    * 브라우저 위에서 사용자와 소통하는 실제 **공연=프로그램**입니다.
    * 이벤트 핸들링(클릭 등)과 데이터 가공(JSON을 예쁜 화면으로 변환)을 담당합니다.

---

## 5. 현대 FE의 트렌드: "CSR과 스켈레톤 UI"

* **CSR (Client-Side Rendering):**
    * 현대 FE의 표준입니다. 서버는 빈 도화지(HTML)와 거대한 붓(JS)만 던져주고, **브라우저가 실시간으로 화면을 그려내는 방식**입니다.
    * 우리는 `.tsx`나 `.jsx`로 코드를 짜지만, 빌드 도구(Vite 등)가 이를 브라우저가 읽을 수 있는 `.js`로 번역하여 전달합니다.
    

* **스켈레톤 UI:**
    * CSR의 '첫 로딩 시 빈 화면'이라는 단점을 보완하는 기법입니다.
    * 데이터가 오기 전까지 **콘텐츠의 회색 윤곽(뼈대)**을 먼저 보여주어 사용자가 기다리는 지루함을 줄여줍니다.
    * **동작 순서:** 접속 → 빈 HTML 로드 → JS 실행 및 **스켈레톤 노출** → API 데이터 도착 → 실제 데이터로 교체.
    

---

> **요약**

> "이 프로젝트는 사용자가 접속하면 빈 종이를 먼저 받고, 우리가 짠 자바스크립트가 실행되면서 화면을 그려내는 **CSR 방식**입니다. 
> 그동안 사용자가 지루하지 않게 **스켈레톤 UI**를 보여줄 것이고, **JWT**라는 신분증으로 보안을 챙기며, **상태 관리**를 통해 모든 화면에서 끊김 없는 경험을 제공할 예정입니다."

## 6. 신분증의 보관과 제시: "금고와 자동 문지기"

로그인 후 받은 신분증(JWT)을 어디에 두고 어떻게 꺼내 쓰는지에 대한 구체적인 메커니즘입니다.

* **보관: Secure Storage (암호화 금고)**
    * **LocalStorage(주머니):** 누구나 접근하기 쉬워 해커가 훔쳐가기 쉽습니다.
    * **Secure Storage(금고):** 하드웨어 수준이나 OS에서 제공하는 보안 영역에 암호화하여 저장합니다. 
      * 우리 프로젝트는 보안을 위해 이 **금고**에 신분증을 보관합니다.
* **제시: Authorization Header (신분증 제시)**
    * 모든 API 요청은 백엔드라는 성문을 통과해야 합니다. 
      * 이때 편지 봉투(HTTP Header)의 특정 칸(`Authorization`)에 신분증을 끼워 보냅니다.
* **인터셉터(Interceptor)의 역할:**
    * 매번 코드로 신분증을 꺼내서 붙이는 것은 번거롭고 실수하기 쉽습니다. 
      * 그래서 모든 요청이 나가기 직전, 자동으로 금고에서 신분증을 꺼내 헤더에 착! 붙여주는 **'자동 신분증 제시 장치'**를 설정하여 사용합니다.

## 7. 렌더링의 흐름: "정적 배달과 동적 변신"

* **정적 렌더링 (Static Rendering):**
    * 서버가 미리 만들어둔 **변하지 않는 파일**들을 보내는 단계입니다. 누가 접속해도 똑같은 빈 도화지(HTML)와 도구(JS)를 받습니다. 매우 빠르지만 내용은 비어 있습니다.
* **동적 렌더링 (Dynamic Rendering):**
    * 브라우저에 도착한 도구(JS)가 실행되면서, 접속한 사용자의 데이터에 맞춰 **실시간으로 화면을 구성**하는 단계입니다.
* **CSR에서의 실행 흐름:**
    1. **배달(Static):** 브라우저가 빈 HTML과 JS 보따리를 받습니다.
    2. **해석(Runtime):** 브라우저 엔진이 JS 보따리를 풀어 로직을 실행합니다.
    3. **대기(Skeleton):** 데이터를 가져오는 동안 뼈대 UI를 먼저 그려 사용자를 안심시킵니다.
    4. **완성(Dynamic):** API를 통해 가져온 데이터로 빈칸을 채워 최종 화면을 완성합니다.

## 8. HTTP 요청의 3대 요소: "API는 단순한 주소가 아니다"

API 호출은 단순히 주소를 부르는 것이 아니라, 규격에 맞는 **우편물**을 보내는 과정입니다.

* **주소(URL)는 목적지일 뿐:** 주소는 편지가 어디로 갈지를 정하는 이정표입니다.
  * 실제 내용은 **HTTP 요청**이라는 패키지에 담깁니다.
* **HTTP 요청의 3대 요소 (우편물 비유):**
    1. **Method (우편물 종류):** 이 편지가 '조회용(GET)'인지, '제출용(POST)'인지, '수정용(PUT)'인지 용도를 결정합니다.
    2. **Headers (봉투 겉면):** 보내는 사람의 신분증(JWT), 내용물의 종류(JSON인지 이미지인지) 등 부가 정보를 적습니다.
    3. **Body (편지 내용물):** 백엔드에 전달할 진짜 데이터입니다. 분석할 이미지 파일이나 입력한 메모 텍스트가 여기에 담깁니다.

## 💡 최종 요약

1. 보안을 위해 신분증을 **암호화 금고**에 숨겨두고, 
2. **인터셉터**라는 자동 장치를 통해 **HTTP 헤더**에 신분증을 끼워 보내며, 
3. 브라우저가 처음 받은 **정적 파일**을 자바스크립트로 **동적**하게 변신시켜 화면을 완성하는 **CSR 기반의 서비스**입니다. 

이 흐름을 이해해야 우리가 왜 스켈레톤 UI를 만들고, 왜 API 통신 라이브러리를 설정하는지 명확히 알 수 있습니다."
